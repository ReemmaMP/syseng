Must haves

(These are the absolute minimum requirements that we will strive to meet during this project. Fulfilling each of these requirements will constitute success for the project.)

Cross-Platform app: it must work on both Android and iOS
Android and iOS are the two major target platforms for this project as they have the greatest market share. In order to make this requirement easier we are allowed to have shared code between the two deployments (e.g. we should create one app that works on both platforms as opposed to two separate apps.)
C# Xamarin code-base
The clients want us to build the application in Xamarin so that we can write the entire app in C# and also export it for multiple platforms easily. This is also because the project is sponsored by Microsoft who own Xamarin, and therefore would like to see it in lieu of any competitors' platforms.
Geolocation features to locate the user
A strong part of this application is geolocation, and where possible this should be done with Microsoft APIs. One such API that might be of assistance to us is th Bing Maps one as we can plot the user's location and overlay nearby tourist attractions. However, due to the Bing Maps API being best suited to corporate customers, the client is happy with us using the Google Maps API if we choose to do so.
Integration with the Recommendation Engine/API
As there is already a machine learning backend and API to expose the data written, we should use this API to supply user data to the app. This does mean that we ourselves do not have to derive any data recommendations; we just need to expose the machine learning algorithms that have already been written. Due to the complexity of this recommendations API it is acceptable to not use it if necessary, as the main focus of this project is the tourism functionality.
Database of Landmarks / Points of Interest
We should use Microsoft Azure to keep track of the landmarks and attractions that will be available through the app.

Should haves

(Once we have implemented the Must Haves, we will focus on these features)

A Clean User Interface
A lot of the mapping interface could look very messy, so we will next look into creating a slick User Interface that is both function and attractive.
Ability to Push Notifications
The inital application will show landmarks that are around and interesting based on the user. The next stage would be to push location-specific, timely alerts to the user. This has to be done carefully to ensure that the notifications do not spam the user, do not come up too late and do not drain the user's battery at an unwieldly speed. Even though Notifications themselves use very little battery, constant GPS tracking does use a lot.
An interface for Kent County Council to add/approve Points of Interest
Instead of just pulling information in from Bing POI, it would be useful for the Council to be able to manually add landmarks to the app without waiting for the data to reach Bing's POI database. This would also allow them to add their own custom metadata and write-ups.
A Weather Checking Function
By using a Weather API we could show the user the weather forecast for each Point of Interest to complete the "tourist experience".

Could haves

(If we have spare time on the project once we have implemented all Must Haves and most Should Haves, we will try to implement these features)

Support for Wearables
Microsoft and Kent Council are interested in seeing if this concept could be brought to devices such as the Apple Watch and Android Wear watches. Even though support for these devices comes in out-of-box SDKs, we would need to write an extra interface for each device type, so this is considered a low priority due to time constraints.
Battery Optimisation
Much of the real-time tracking of the user required for location-based push notifications could, if implemented naively, cause rampant battery drain (similar to Pokémon Go). In this part of the project we would consider how to deal with these issues using intelligent algorithms and techniques such as geofencing.
Be an extension of the existing project
Due to the fact that we have had many issues compiling and using the hsConsumer application which was built last year (e.g. the Spot Markets shopping recommendation application), we have pushed down the priority of reusing and/or extending this code as it would not be testable. However, should we find or be given solutions to these issues, we will reconsider writing all our code as an extension to what already exists.
A method of tuning the frequency of the push notifications received by the user
By default if we implemented push notifications we would set it up to notify the user every time they pass something of interest. However, this may not be preferable if the user is in an area densely populated with attractions so they may wish to tune the frequency of these messages. Adding of this functionality would be done at this stage of the project.
Recommendation feature that can be used to plan ahead
This would be a feature in the application that could assist users in planning walks or treks to see the landmarks in the local area. This could be computationally complex to solve intelligently, and therefore is considered mostly out of scope of the project. However, should the recommendation engine yield lots of useful data that could feed into this, it may be possible for us to implement it.
Calendar Integration
One client suggestion was that we hook into the user's calendar to decide or derive when they would be busy (e.g. during meetings or work hours). From this, the app could switch off its notifications so as not to disturb. Due to the multitude of calendar applications around, coupled with the wide array of devices available on the market, this has been reduced in importance as the Return on Investment (time input vs. functional yield here) may not make it viable.
Check + decrease/increase radius depending on location + no of points of interest
This feature could intelligently set the radius of the POIs sent to the user based on how densely populated the area is. For example, if the user is in a particularly historically relevant site then it makes sense to reduce the search radius to just the artefacts around as there likely will be many. However, if they are quite far away from POIs then it makes sense to tell the user about the ones closest, even if they are further away than normal. The decision-making process for this is potentially computationally complex, and would require a lot of manual tuning, so it has been left to the end.

Won't (Necessarily) Have

(These features were either discussed with the clients or discussed internally as a group but, due to scope and time restrictions, they are not expected to feature in our project's submission)

Bot integration to make it more interactive
We debated the idea of having a bot inside the app to have a conversation with the user about what kinds of attractions they would want to see, but we considered it too far out of scope.
A short description of the attraction expands into a full page of information
This feature would provide a slick transition between a short sentence of information into a full page write-up. We considered the complexity of UI work involved in this process and concluded that it would not be worth implementing. The cost/benefit analysis of this implementation did not make it viable, given that this app is a prototype anyway.
Integration with Beacons
We decided that this project would overlap too much with Spot Market Team 2's work, and that we can still accomplish what we want to achieve without support for Bluetooth Beacons.
Any Additional Machine Learning
Machine Learning and bot use of it is being handled by Spot Markets Team 1, so we considered this both too complex and out of the scope of our project.
Windows Phone Support
The market share of Windows Phone does not currently warrant us spending extra time on the project to make it Windows Phone compatible.

Changes in MoSCoW report

The initial set of MoSCoW requirements were made based on our understanding of the clients' requests; we then re-consulted our clients to fix upon these requirements. We also took the user's perspective into consideration through heuristically evaluating and users' interactions with the prototype of the app. Through the various iterations, we were able to improve user experience at each stage.

  • Our initial brief was to create a tourist 'layer' which gave us the option to create two things: an app or a bot. After conducting a heavy amount of research, through weighing the pros and cons of each, we decided to create the app. This was mainly because a bot cannot get the user's location automatically; it must be sent each time as a chat message, which would slow the experience down.

  • Initially, we were planning to use the Spot Markets Recommendation Engine and build on top of it. However, since we ended up spending too much time trying to get it to work and the sample data available was not complete enough for our purposes, we had to move on from it and focus on actually creating an app that would fulfill the remaining requirements.

  • Windows Phone support was initially included in the 'Could Haves' section. It was decided that even though Xamarin would support Windows Phone 10 and above (using UWP), we would not focus on any platform-specific code for it. This was both because of the market share of Windows Phone being so low, and our clients wanting us to focus mainly on Android and iOS.

  • Initially we planned to build on top of work produced by previous students. Due to the difficulty in transition and lack of clarity in the documentation provided, we agreed to keep the projects independent until we could produce a sufficient amount of our own work. At this point we would consider integrating the existing projects.
